# PRD 生成器

创建清晰、可操作、适合实施的产品需求文档。

## 语言要求

**所有输出必须使用中文**，包括：
- 澄清问题及选项
- PRD 所有章节内容（介绍、目标、用户故事、功能需求等）
- 用户故事的描述和验收标准
- 文件名保持英文 kebab-case（如 `prd-task-priority.md`）
- 章节标题使用中文（如 "## 介绍"、"## 用户故事"）

---

## 工作流程

1. 接收用户的功能描述
2. **头脑风暴**：尝试调用 `brainstorming` skill 进行创意发散（如可用）
3. **初步澄清**：使用 `AskUserQuestion` 工具提出 3-5 个关键问题（交互式选择）
4. **代码库探索与约束发现**：自动扫描相关代码，产出结构化约束集合
5. **需求讨论（必选）**：基于约束集合与用户深入讨论细节，补充遗漏，对齐预期
6. 生成结构化 PRD
7. 保存到 `tasks/prd-[feature-name].md`

**重要：** 不要开始实现代码。只创建 PRD。

---

## 第一步：头脑风暴

在收到用户的功能描述后，**先尝试调用 `superpowers:brainstorming` skill**（通过 Skill 工具）进行创意发散。

如果该 skill 不可用，则自行完成以下思考并输出给用户：

- 这个功能可以有哪些实现方向？（列出 2-3 种思路）
- 类似产品是怎么做的？有哪些值得借鉴的设计？
- 用户真正的痛点是什么？表面需求背后有没有更深层的问题？
- 有没有更简单的方案能达到 80% 的效果？

将头脑风暴的关键结论作为后续问题和 PRD 编写的输入。

---

## 第二步：初步澄清

针对模糊或缺失的信息，使用 **`AskUserQuestion` 工具**向用户提问。

### 提问要求

- 每次调用提 1-4 个问题
- 每个问题提供 2-4 个选项，用户通过空格勾选、回车提交
- 支持 `multiSelect: true` 让用户多选
- 用户始终可以选择"Other"输入自定义内容
- 问题聚焦于以下维度：
  - **问题/目标：** 这个功能解决什么问题？
  - **核心功能：** 关键操作有哪些？
  - **范围/边界：** 不做什么？
  - **成功标准：** 怎么判断完成了？

### 调用示例

```json
{
  "questions": [
    {
      "question": "这个功能的主要目标是什么？",
      "header": "目标",
      "multiSelect": false,
      "options": [
        { "label": "改善引导体验", "description": "优化新用户首次使用的引导流程" },
        { "label": "提高用户留存", "description": "通过功能增强让用户持续使用" },
        { "label": "减少客服负担", "description": "通过自助功能降低人工支持需求" }
      ]
    },
    {
      "question": "目标用户是谁？",
      "header": "用户",
      "multiSelect": true,
      "options": [
        { "label": "新用户", "description": "注册后尚未深度使用的用户" },
        { "label": "活跃用户", "description": "已在日常使用产品的用户" },
        { "label": "管理员", "description": "有管理权限的内部用户" }
      ]
    },
    {
      "question": "实现范围是什么？",
      "header": "范围",
      "multiSelect": false,
      "options": [
        { "label": "最小可用版本", "description": "只做核心功能，快速验证" },
        { "label": "完整实现", "description": "包含所有计划中的功能点" },
        { "label": "仅后端/API", "description": "只做数据和接口层" },
        { "label": "仅前端 UI", "description": "只做界面展示层" }
      ]
    }
  ]
}
```

可根据需要多次调用 `AskUserQuestion`，直到关键信息收集完整。

## 第三步：代码库探索与约束发现

**在进入需求讨论之前，先自动扫描代码库**，让后续讨论建立在真实的技术基础上，而非纯粹的猜测。

### 探索方法

使用 `Explore` 类型的 Agent 按**上下文边界**（而非角色）并行扫描相关代码区域：

- **按上下文边界分割**，例如："用户相关模块代码"、"路由与 API 逻辑"、"配置与基础设施"
- **禁止按角色分割**（如"前端专家"、"后端专家"）——每个探索边界应自包含
- 如果项目较小（单目录），可以用单个 Agent 完成

### 每个探索边界的产出模板

```
探索边界: [被探索的上下文区域]
已有结构: [发现的关键模式和组件]
已有约定: [正在使用的规范和惯例]
硬约束:   [技术限制，不可违反的模式]
软约束:   [惯例、偏好、风格指南]
依赖关系: [跨模块的依赖]
可复用资源: [已有的组件/工具/模块可直接复用]
风险:     [潜在阻碍]
```

### 汇总约束集合

将所有探索结果合并为统一的约束集合，分为四类：

1. **硬约束**：技术限制、不可违反的既有模式（如"项目使用 Next.js App Router，不能用 Pages Router"）
2. **软约束**：惯例、偏好（如"现有组件统一使用 shadcn/ui"）
3. **依赖关系**：影响实现顺序的跨模块关系
4. **风险**：需要提前规避的阻碍

将约束集合输出给用户，作为下一步需求讨论的输入。

---

## 第四步：需求讨论（强制）

**此步骤不可跳过。** 在收集完初步信息和代码库约束后，必须与用户进行至少一轮深入的需求讨论。

### 讨论内容

基于前三步收集的信息（头脑风暴 + 用户澄清 + 代码库约束），整理并输出以下内容，逐项与用户确认：

1. **需求摘要**：用 3-5 句话概括你理解的需求全貌
2. **核心用户流程**：描述用户使用该功能的主要路径（步骤 1 → 步骤 2 → ...）
3. **边界情况**：列出你识别到的边界场景和异常情况，询问用户期望的处理方式
4. **技术影响评估**：该功能可能影响哪些现有模块？是否有潜在冲突？
5. **补充问题**：基于以上分析提出更细粒度的问题

### 讨论格式

```
## 需求理解确认

### 我的理解
[用 3-5 句话描述需求全貌]

### 核心用户流程
1. 用户进入 XX 页面
2. 点击 XX 按钮
3. 系统执行 XX
4. 用户看到 XX 反馈

### 代码库约束摘要
- 硬约束：[不可违反的技术限制]
- 软约束：[建议遵循的惯例]
- 可复用：[已有组件/模块可直接使用]
- 风险：[需提前规避的问题]

### 我识别到的边界情况
- 场景 A：[描述] → 建议处理方式：[方案]
- 场景 B：[描述] → 需要你确认：[问题]

### 技术影响
- 需要修改模块：[列表]
- 可能影响的现有功能：[列表]

### 补充问题
```

然后使用 `AskUserQuestion` 工具对补充问题进行交互式确认。

**讨论可进行多轮**，直到双方对需求细节达成一致。当用户确认无更多补充时，才进入 PRD 编写阶段。

---

## 第五步：编写 PRD

基于头脑风暴、澄清问题、代码库约束和需求讨论的全部成果，生成包含以下章节的 PRD：

### 1. 介绍/概述
简要描述功能及其解决的问题。

### 2. 目标
具体、可衡量的目标（列表形式）。

### 3. 用户故事
每个故事需要：
- **标题：** 简短描述性名称
- **描述：** "作为[用户角色]，我希望[功能]，以便[收益]"
- **验收标准：** 可验证的完成清单

每个故事应小到可以在一次集中开发中完成。

```markdown
### US-001: [标题]
**描述：** 作为[用户角色]，我希望[功能]，以便[收益]。

**验收标准：**
- [ ] 具体可验证的标准
- [ ] 另一个标准
- [ ] 类型检查/lint 通过
- [ ] **[仅 UI 故事]** 使用 dev-browser skill 在浏览器中验证
```

**重要：**
- 验收标准必须可验证，不能模糊。"正常工作"是不好的。"点击按钮后显示删除确认对话框"是好的。
- **任何涉及 UI 变更的故事：** 必须包含"使用 dev-browser skill 在浏览器中验证"作为验收标准。

### 4. 功能需求
编号的具体功能列表，每条需求关联验证方法：
- "FR-1: 系统必须允许用户..." → 验证：[如何验证此需求已满足]
- "FR-2: 当用户点击 X 时，系统必须..." → 验证：[具体验证方式]

表述要明确、无歧义。每条功能需求都应有对应的验证方法，不只是用户故事有验收标准。

### 5. 非目标（不在范围内）
本功能不包含的内容。对范围管理至关重要。

### 6. 设计考虑（可选）
- UI/UX 需求
- 设计稿链接（如有）
- 可复用的现有组件

### 7. 技术约束（基于代码库探索）
从第三步代码库探索中提炼的约束集合：
- **硬约束**：技术限制、不可违反的既有模式
- **软约束**：惯例、偏好、风格指南
- **依赖关系**：影响实现顺序的跨模块关系
- **可复用资源**：已有组件/工具/模块可直接复用
- **风险与规避**：潜在阻碍及建议的规避策略

如果第三步未发现有意义的约束（如纯新项目），可标注"无已知技术约束"。

### 8. 成功指标
如何衡量成功？
- "将完成 X 的时间减少 50%"
- "将转化率提高 10%"

### 9. 待解决问题
需要进一步澄清的问题或领域。

---

## 面向初级开发者编写

PRD 读者可能是初级开发者或 AI agent。因此：

- 表述明确、无歧义
- 避免使用术语，或对术语进行解释
- 提供足够的细节来理解目的和核心逻辑
- 对需求进行编号以便引用
- 在有帮助的地方使用具体示例

---

## 输出

- **格式：** Markdown (`.md`)
- **位置：** `tasks/`
- **文件名：** `prd-[feature-name].md`（kebab-case）

## 示例 PRD

```markdown
# PRD：任务优先级系统

## 介绍

为任务添加优先级，让用户能专注于最重要的事项。任务可标记为高、中、低优先级，通过视觉标识和筛选帮助用户有效管理工作量。

## 目标

- 允许为任何任务分配优先级（高/中/低）
- 提供清晰的优先级视觉区分
- 支持按优先级筛选和排序
- 新任务默认为中优先级

## 用户故事

### US-001: 在数据库中添加优先级字段
**描述：** 作为开发者，我需要在数据库中存储任务优先级，以便跨会话持久化。

**验收标准：**
- [ ] 在 tasks 表中添加 priority 列：'high' | 'medium' | 'low'（默认 'medium'）
- [ ] 成功生成并运行迁移
- [ ] 类型检查通过

### US-002: 在任务卡片上显示优先级标识
**描述：** 作为用户，我希望一眼看到任务优先级，以便知道哪些需要优先处理。

**验收标准：**
- [ ] 每张任务卡片显示彩色优先级徽标（红=高，黄=中，灰=低）
- [ ] 无需悬停或点击即可看到优先级
- [ ] 类型检查通过
- [ ] 使用 dev-browser skill 在浏览器中验证

### US-003: 在任务编辑中添加优先级选择器
**描述：** 作为用户，我希望在编辑任务时能修改优先级。

**验收标准：**
- [ ] 任务编辑弹窗中包含优先级下拉框
- [ ] 显示当前优先级为已选状态
- [ ] 选择后立即保存
- [ ] 类型检查通过
- [ ] 使用 dev-browser skill 在浏览器中验证

### US-004: 按优先级筛选任务
**描述：** 作为用户，我希望能筛选任务列表，在专注时只看到高优先级任务。

**验收标准：**
- [ ] 筛选下拉框选项：全部 | 高 | 中 | 低
- [ ] 筛选状态持久化在 URL 参数中
- [ ] 无匹配任务时显示空状态提示
- [ ] 类型检查通过
- [ ] 使用 dev-browser skill 在浏览器中验证

## 功能需求

- FR-1: 在 tasks 表中添加 `priority` 字段（'high' | 'medium' | 'low'，默认 'medium'） → 验证：迁移成功运行，字段存在且默认值正确
- FR-2: 在每张任务卡片上显示彩色优先级徽标 → 验证：浏览器中可见红/黄/灰徽标
- FR-3: 在任务编辑弹窗中包含优先级选择器 → 验证：修改后刷新页面数据持久化
- FR-4: 在任务列表头部添加优先级筛选下拉框 → 验证：筛选后列表只显示对应优先级
- FR-5: 在每个状态列内按优先级排序（高到低） → 验证：创建不同优先级任务，观察排序

## 非目标

- 不实现基于优先级的通知或提醒
- 不实现基于截止日期的自动优先级分配
- 不实现子任务的优先级继承

## 技术约束

- **硬约束**：使用现有数据库 ORM，不引入新的数据层
- **软约束**：复用现有的徽标组件并扩展颜色变体
- **依赖关系**：FR-1（数据库字段）必须先于 FR-2~FR-5 实现
- **可复用资源**：现有 Badge 组件、URL search params 工具函数
- **风险**：筛选状态通过 URL search params 管理，需确认与现有路由参数不冲突

## 成功指标

- 用户可以在 2 次点击内修改优先级
- 高优先级任务在列表顶部立即可见
- 任务列表性能无退化

## 待解决问题

- 优先级是否应影响列内任务排序？
- 是否需要添加优先级快捷键？
```

## 保存前检查清单

保存 PRD 前确认：

- [ ] 已尝试调用 brainstorming skill（或自行完成头脑风暴）
- [ ] 已使用 AskUserQuestion 工具进行交互式澄清
- [ ] 已完成代码库探索，产出约束集合（或确认无需探索）
- [ ] 已完成至少一轮需求讨论，用户确认无补充
- [ ] 已整合所有讨论成果和约束集合到 PRD 中
- [ ] 用户故事小而具体
- [ ] 功能需求有编号、无歧义，且每条关联验证方法
- [ ] 技术约束章节包含硬约束/软约束/依赖/可复用资源/风险
- [ ] 非目标章节定义了清晰的边界
- [ ] 已保存到 `tasks/prd-[feature-name].md`
